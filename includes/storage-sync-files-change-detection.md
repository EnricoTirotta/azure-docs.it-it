Le modifiche apportate alla condivisione file di Azure mediante il portale di Azure, SMB o REST non vengono rilevate e replicate immediatamente come le modifiche all'endpoint server. Le modifiche non vengono replicate immediatamente perché nel journal File di Azure non sono ancora disponibili notifiche o inserimenti per le modifiche stesse, pertanto non è possibile avviare automaticamente una sessione di sincronizzazione quando i file vengono modificati. In Windows Server Sincronizzazione file di Azure usa [l'inserimento nel journal del numero di sequenza di aggiornamento (USN) di Windows](https://msdn.microsoft.com/library/windows/desktop/aa363798.aspx) per avviare automaticamente una sessione di sincronizzazione quando i file vengono modificati.

Per rilevare le modifiche alla condivisione file di Azure, Sincronizzazione file di Azure include il processo pianificato denominato "processo di rilevamento modifiche", che enumera ogni file nella condivisione e lo confronta con la versione di sincronizzazione del file stesso. Se il processo di rilevamento modifiche determina che i file sono stati modificati, Sincronizzazione file di Azure avvia una sessione di sincronizzazione. Il processo di rilevamento modifiche viene avviato ogni 24 ore. Poiché il processo di rilevamento modifiche funziona mediante l'enumerazione di tutti i file nella condivisione file di Azure, gli spazi dei nomi di grandi dimensioni richiedono più tempo rispetto agli spazi dei nomi più piccoli. Per gli spazi dei nomi di grandi dimensioni il tempo necessario per determinare quali file sono stati modificati può risultare superiore a 24 ore. 

In futuro è prevista l'aggiunta di una funzionalità di rilevamento modifiche in File di Azure simile al numero di sequenza di aggiornamento (USN) in Windows Server. È possibile incrementare la priorità dello sviluppo futuro di questa funzionalità votandola in [UserVoice per File di Azure](https://feedback.azure.com/forums/217298-storage/category/180670-files).
